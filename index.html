How it works & troubleshooting

Flow. Desktop (receiver) creates a WebRTC offer and shows it as a QR. Phone (sender) scans it, builds the answer, shows that as a QR. Desktop's webcam scans the answer. The connection is now established and the IDB data streams directly between browsers — no server in the middle.

QR chunks. WebRTC handshake data is too big for one QR code (typically 3-5 KB), so it's split into multiple QR codes that cycle automatically. Cameras stay on until all chunks are received in any order.

Same origin required. Both devices must open this page from the same URL. IndexedDB is partitioned per origin.

Needs internet. WebRTC uses public STUN servers (Google's free ones) to discover network paths. Tiny amount of traffic, but a working internet connection is required on both ends.

If connection fails: typically a strict firewall blocking WebRTC. Reload both pages and start over. Some corporate networks block this entirely.
