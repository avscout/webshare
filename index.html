How sessions work

Session model. Each session has a stable ID like idb-tomato-rabbit-42, a user-editable name, created/modified timestamps, and a data object holding records keyed by their own IDs (e.g. { item_001: {...}, item_002: {...} }). All sessions live in one local IndexedDB called webshare-sessions.

The session ID never changes — it's generated once when the session is created. This is how both devices know "this incoming session is the same one I already have, not a new one."

Two re-transfer strategies. Replace with confirmation overwrites the local copy after asking. Merge — phone wins combines incoming records into the local session, phone's values overwriting on overlap. Pick what fits your workflow.

Backends. PeerJS uses their free signaling cloud — works across networks. Raw WebRTC is fully peer-to-peer via QR handshake but needs both devices on the same Wi-Fi.
